# Техническое задание

## на разработку десктопной программы для автоматического заполнения юридических документов по шаблонам Word

## 1\. Общие положения

### 1.1. Наименование системы

Десктопная программа для автоматического заполнения юридических документов по шаблонам Word на основе карточек проектов.

### 1.2. Цель разработки

Необходимо разработать десктопную программу, которая позволяет:

1. хранить и использовать библиотеку шаблонов документов;
2. работать с карточками проектов;
3. автоматически определять переменные, используемые в выбранных шаблонах;
4. сопоставлять переменные шаблонов с полями карточки проекта;
5. выявлять недостающие значения;
6. предлагать пользователю заполнить недостающие поля;
7. сохранять введенные значения обратно в карточку проекта;
8. формировать готовые документы Word по выбранным шаблонам;
9. сохранять сформированные документы в указанную пользователем папку.

### 1.3. Границы первой версии

Первая версия программы должна быть рабочим заполнителем документов в формате `.docx`.

В первую версию не входят:

1. полноценная поддержка формата `.doc`;
2. фоновая синхронизация в реальном времени;
3. сложная логика справочников со множественными связями;
4. расширенное ручное конфигурирование карточек шаблонов;
5. предварительный просмотр документа;
6. многопользовательский режим;
7. журнал истории изменений.

---

## 2\. Основная модель системы

### 2.1. Основные сущности

Система должна включать следующие основные сущности:

1. проект;
2. карточка проекта;
3. шаблон документа;
4. карточка шаблона;
5. словарь переменных;
6. справочники;
7. настройки программы.

### 2.2. Логические группы полей

В архитектуре системы должны быть разделены следующие группы полей:

#### 2.2.1. Поля проекта

Это данные, относящиеся к конкретному делу или проекту:

1. реквизиты заказчика;
2. реквизиты должника;
3. реквизиты цедента;
4. судебные реквизиты;
5. финансовые данные;
6. иные данные, используемые в документах по данному делу.

#### 2.2.2. Поля шаблона

Это перечень переменных, которые программа извлекает из текста конкретного шаблона документа.

#### 2.2.3. Служебные поля

Это поля, используемые системой для внутренней работы:

1. путь к шаблону;
2. категория шаблона;
3. правило имени файла;
4. путь выгрузки;
5. дата формирования;
6. статус активности шаблона;
7. признаки обязательности полей;
8. технические коды переменных.

---

## 3\. Источники данных

### 3.1. Источник проектов

Источником проектов является Excel-файл, расположенный на локальном диске пользователя и, как правило, синхронизируемый через Яндекс Диск, Google Диск или аналогичное облачное хранилище.

### 3.2. Требования к Excel-файлу проектов

1. Путь к Excel-файлу задается в разделе «Настройки».
2. Программа должна уметь загружать проекты из Excel-файла.
3. Программа должна уметь по команде пользователя обновлять данные из Excel-файла.
4. Программа должна уметь сохранять изменения, внесенные в программе, обратно в Excel-файл.
5. Перед записью изменений программа должна создавать резервную копию Excel-файла.
6. На первом этапе синхронизация выполняется только по явным действиям пользователя:
	1. открыть;
	2. обновить;
	3. сохранить.

### 3.3. Источник шаблонов

Источником шаблонов является папка на локальном диске пользователя, содержащая `.docx`\-файлы, при необходимости синхронизируемая через облачное хранилище.

### 3.4. Требования к библиотеке шаблонов

1. Путь к библиотеке шаблонов задается в разделе «Настройки».
2. Программа должна сканировать указанную папку и все вложенные подпапки.
3. Программа должна сохранять иерархию папок при отображении шаблонов.
4. Программа должна по кнопке «Обновить шаблоны» повторно сканировать библиотеку.
5. Для каждого найденного шаблона программа должна автоматически создавать или обновлять карточку шаблона.

---

## 4\. Форматы документов

### 4.1. Поддерживаемые форматы первой версии

Первая версия программы должна поддерживать только формат `.docx`.

### 4.2. Ограничение по формату `.doc`

Поддержка `.doc` в первую версию не включается. Если в библиотеке имеются старые файлы `.doc`, они должны предварительно конвертироваться пользователем в `.docx`.

## 5\. Структура интерфейса первой версии

### 5.1. Основные разделы интерфейса

Программа должна содержать 4 основных раздела:

1. «Проекты»;
2. «Шаблоны»;
3. «Заполнение»;
4. «Настройки».

### 5.2. Минимальный интерфейс первой версии

Первая версия интерфейса должна включать:

1. список проектов;
2. просмотр и редактирование карточки проекта;
3. список шаблонов;
4. просмотр карточки шаблона;
5. окно выбора шаблонов;
6. окно недостающих полей;
7. кнопку формирования документов;
8. раздел настроек путей;
9. кнопки обновления проектов и шаблонов.

## 6\. Раздел «Проекты»

### 6.1. Назначение раздела

Раздел «Проекты» предназначен для:

1. отображения списка проектов;
2. выбора проекта;
3. открытия карточки проекта;
4. редактирования карточки проекта;
5. сохранения изменений в Excel-файл;
6. обновления списка проектов из Excel.

### 6.2. Карточка проекта

Карточка проекта представляет собой анкету с реквизитами и значениями переменных, относящимися к конкретному судебному делу или проекту.

### 6.3. Обязательные действия в разделе

Пользователь должен иметь возможность:

1. открыть карточку проекта;
2. просмотреть значения полей;
3. перейти в режим редактирования;
4. изменить значения полей;
5. добавить новое поле;
6. сохранить изменения.

### 6.4. Поддержка типов полей

На первом этапе должны поддерживаться:

1. текстовое поле;
2. многострочное текстовое поле;
3. выпадающий список для ограниченного числа справочных полей.

### 6.5. Выпадающие списки первой версии

В первой версии допускается реализация ограниченного числа справочников, например:

1. юрист;
2. подписант.

При выборе значения из справочника программа должна автоматически подставлять связанные данные, если они предусмотрены настройкой справочника.

### 7.2. Приведение дублирующихся и неоднородных переменных к единообразию

В первой версии должен быть принят единый словарь названий. Ниже — рекомендованная нормализация на уровне отображаемых имен:

1. `{ИНН Должник}` и `{ИНН ДОЛЖНИКА}` → `ИНН должника`
2. `{ОГРН Должник}` и `{ОГРН Должника}` → `ОГРН должника`
3. `{Юр Адрес должника}` и `{Юр. адрес Должник}` → `Юр. адрес должника`
4. `{дата решения}` и `{Дата решения}` → `Дата решения`
5. `{Должник}` и `{ДОЛЖНИК}` → `Должник`
6. `{Какого Арбитражного суда?}` и `{Арб. Суд}` при необходимости должны быть разведены по смыслу:
	1. если оба поля фактически означают одно и то же — использовать `Наименование арбитражного суда`;
	2. если одно поле хранит краткое название, а другое — форму для вставки в текст, они должны быть разделены как два разных поля:
		1. `Арб. суд (кратко)`;
		2. `Арбитражный суд (для вставки в текст)`.
7. `{по оплате(вид услуг)}` → `Вид услуг`
8. `{Цедент П.Наим}` → `Цедент полное наименование`
9. `{цена уступки}` → `Цена уступки`
10. `{цена прописью}` → `Цена прописью`
11. `{к/сч. Банка Заказчика}` → `Корреспондентский счет банка заказчика`
12. `{р/сч. Заказчика}` → `Расчетный счет заказчика`
13. `{Юр Адрес Заказчика}` → `Юр. адрес заказчика`
14. `{Юр. адрес цедента}` → `Юр. адрес цедента`
15. `{Номер Цессии}` → `Номер цессии`
16. `{Резюлютивка}` и `{резолютивка – взыскать с}` не должны автоматически считаться одним полем, так как по смыслу это могут быть разные данные:
	1. `Резолютивка`;
	2. `Резолютивка – взыскать с`.

### 7.3. Правило первой версии по совпадению полей

В первой версии система должна использовать словарь переменных и сопоставлять поля через нормализованные названия, а не только по буквальному совпадению текста.

## 8\. Словарь переменных

### 8.1. Назначение словаря

Словарь переменных должен быть отдельным компонентом системы и использоваться для:

1. унификации названий переменных;
2. сопоставления полей Excel, карточек проектов и шаблонов;
3. устранения дублирующихся обозначений;
4. копирования переменных по одной и списком;
5. дальнейшего расширения и нормализации системы.

### 8.2. Структура записи словаря

Для каждой переменной словарь должен хранить как минимум:

1. техническое имя;
2. отображаемое имя;
3. исходные варианты написания;
4. тип поля;
5. принадлежность к группе:
	1. поле проекта;
	2. поле шаблона;
	3. служебное поле;
6. признак обязательности;
7. комментарий.

### 8.3. Функции словаря первой версии

В первой версии словарь должен позволять:

1. собирать переменные из шаблонов;
2. собирать поля из Excel-файла проектов;
3. отображать единый список переменных;
4. копировать одну переменную;
5. копировать список переменных целиком.

---

## 9\. Раздел «Шаблоны»

### 9.1. Назначение раздела

Раздел «Шаблоны» предназначен для:

1. просмотра библиотеки шаблонов;
2. отображения структуры папок;
3. просмотра карточки шаблона;
4. обновления библиотеки шаблонов;
5. автоматического формирования карточек шаблонов.

### 9.2. Категории шаблонов

Категория шаблона должна определяться по пути к файлу в библиотеке шаблонов.

Примеры:

1. `Банкротство/Заявление о банкротстве.docx`
2. `Претензии/Претензия должнику.docx`

### 9.3. Карточка шаблона

Карточка шаблона должна содержать:

1. наименование шаблона;
2. путь к файлу;
3. категорию;
4. список переменных, найденных в тексте;
5. порядок переменных по первому появлению в тексте;
6. список переменных без повторов;
7. правило формирования имени выходного файла;
8. признак активности шаблона.

### 9.4. Правило определения переменных в шаблоне

Переменной считается любой фрагмент текста в фигурных скобках вида:

Plain text

{Переменная}

Такой подход соответствует присланному пользователем шаблону.

### 9.5. Правила обновления шаблонов

По кнопке «Обновить шаблоны» программа должна:

1. заново сканировать библиотеку шаблонов;
2. обнаруживать новые шаблоны;
3. обновлять карточки измененных шаблонов;
4. пересчитывать список переменных;
5. сохранять структуру папок и категорий.

---

## 10\. Раздел «Заполнение»

### 10.1. Назначение раздела

Раздел «Заполнение» предназначен для:

1. выбора проекта;
2. выбора одного или нескольких шаблонов;
3. формирования списка недостающих полей;
4. заполнения недостающих значений;
5. запуска формирования документов.

### 10.2. Логика работы

Работа раздела должна происходить в следующем порядке:

1. пользователь выбирает проект;
2. пользователь выбирает один или несколько шаблонов;
3. программа извлекает все переменные выбранных шаблонов;
4. программа сопоставляет их с карточкой проекта через словарь переменных;
5. программа формирует список недостающих полей;
6. программа показывает окно заполнения недостающих полей;
7. пользователь вводит недостающие значения;
8. введенные значения сохраняются в карточку проекта;
9. после заполнения всех обязательных полей активируется кнопка «Заполнить шаблоны»;
10. после нажатия кнопки программа формирует документы.

### 10.3. Представление полей в разделе «Заполнение»

В интерфейсе должны быть отдельно показаны:

1. уже заполненные поля;
2. поля, требующие заполнения.

Это требование является обязательным для первой версии.

### 10.4. Общая анкета по нескольким шаблонам

При выборе нескольких шаблонов программа должна:

1. объединить переменные всех выбранных шаблонов;
2. убрать повторы;
3. сопоставить их с карточкой проекта;
4. вывести пользователю только недостающие или пустые поля.

---

## 11\. Формирование документов

### 11.1. Общие требования

После нажатия кнопки «Заполнить шаблоны» программа должна:

1. взять выбранный проект;
2. взять выбранные шаблоны;
3. подставить значения всех переменных;
4. сформировать готовые `.docx`\-документы;
5. сохранить документы в указанную папку выгрузки.

### 11.2. Формирование имен файлов

Для каждого шаблона должно быть возможно задать правило формирования имени выходного файла.

Пример:

Plain text

{%filename%} - {ЗАКАЗЧИК}.{ДОЛЖНИК}

### 11.3. Требования к именам файлов

Программа должна:

1. подставлять значения переменных в имя файла;
2. очищать имя файла от недопустимых символов;
3. предотвращать перезапись существующих файлов;
4. при совпадении имен добавлять суффикс или порядковый номер.

---

## 12\. Раздел «Настройки»

### 12.1. Раздел должен позволять задавать:

1. путь к Excel-файлу проектов;
2. путь к библиотеке шаблонов;
3. путь к папке выгрузки документов.

### 12.2. В разделе должны быть доступны команды:

1. обновить проекты;
2. обновить шаблоны;
3. проверить доступность путей;
4. создать резервную копию Excel-файла проектов перед сохранением.

---

## 13\. Резервное копирование

### 13.1. Общие требования

Перед каждой записью изменений в Excel-файл проектов программа должна автоматически создавать резервную копию.

### 13.2. Минимальные требования к резервной копии

1. резервная копия должна сохраняться в отдельную подпапку;
2. имя резервной копии должно содержать дату и время;
3. пользователь должен иметь возможность восстановить исходный файл вручную.

---

## 14\. Обработка ошибок

Программа должна корректно обрабатывать и понятно сообщать пользователю о следующих ситуациях:

1. не указан путь к Excel-файлу проектов;
2. не указан путь к библиотеке шаблонов;
3. не указан путь к папке выгрузки;
4. Excel-файл недоступен;
5. библиотека шаблонов недоступна;
6. шаблон поврежден;
7. шаблон не удалось прочитать;
8. в шаблоне есть переменные, которым не сопоставлены значения;
9. пользователь не выбрал проект;
10. пользователь не выбрал ни одного шаблона;
11. не удалось сохранить готовый документ;
12. имя файла содержит недопустимые символы;
13. не удалось создать резервную копию;
14. в Excel-файле обнаружены несопоставимые или некорректные столбцы.

## 15\. Минимальный функционал первой версии

Первая версия должна обязательно обеспечивать:

1. загрузку проектов из Excel-файла;
2. просмотр карточки проекта;
3. редактирование карточки проекта;
4. сохранение изменений в Excel-файл;
5. резервное копирование Excel-файла перед записью;
6. загрузку библиотеки шаблонов `.docx`;
7. сканирование подпапок библиотеки;
8. автоматическое извлечение переменных из шаблонов;
9. создание карточек шаблонов;
10. категоризацию шаблонов по структуре папок;
11. выбор одного проекта;
12. выбор одного или нескольких шаблонов;
13. формирование списка недостающих полей;
14. отдельное отображение заполненных и незаполненных полей;
15. дозаполнение недостающих полей;
16. сохранение введенных значений в карточку проекта;
17. формирование документов;
18. сохранение документов в указанную папку;
19. базовый словарь переменных;
20. копирование переменных по одной и списком.

## 16\. Этапность разработки

### 16.1. Этап 1

Рабочий заполнитель документов `.docx`:

1. проекты из Excel;
2. шаблоны из папки;
3. извлечение переменных;
4. словарь переменных;
5. общая анкета;
6. формирование документов.

### 16.2. Этап 2

Развитие словаря переменных:

1. расширенная нормализация;
2. ручная корректировка сопоставлений;
3. улучшение карточек шаблонов.

### 16.3. Этап 3

Справочники и поля со списками:

1. юристы;
2. подписанты;
3. автоподстановка связанных данных.

### 16.4. Этап 4

Улучшение удобства и устойчивости:

1. дополнительные настройки;
2. диагностика ошибок;
3. расширение интерфейса;
4. дополнительные режимы работы.

## 17\. Результат первой версии

Результатом первой версии должна быть десктопная программа, которая позволяет:

1. открыть проекты из Excel;
2. открыть библиотеку шаблонов `.docx`;
3. выбрать проект;
4. выбрать шаблоны;
5. автоматически определить недостающие поля;
6. заполнить недостающие значения;
7. сохранить их в карточке проекта;
8. сформировать готовые документы в выбранную папку.
